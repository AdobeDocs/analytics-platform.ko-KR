# GA4와 Customer Journey Analytics 데이터가 다른 이유

GA4와 Customer Journey Analytics은 동일한 기간 및 동일한 명확한 지표에 대해 서로 다른 숫자를 보고하는 것이 일반적입니다. 다양한 데이터 수집 방법, 지표 정의, ID 모델 및 세션 규칙이 모두 불일치에 기여합니다. 이 페이지에서는 가장 일반적인 차이점을 설명하고 설명되지 않은 차이를 감사하기 위한 지침을 제공합니다.

## 참여 세션

GA4는 세션이 10초 이상 지속되거나, 하나 이상의 키 이벤트를 포함하거나, 페이지 보기가 2개 이상인 경우 세션을 **참여**(으)로 처리합니다. 이 단일 정의는 참여 비율, 바운스 비율 및 활성 사용자 뒤의 참여 임계값을 포함한 여러 GA4 지표의 밑바탕에 적용됩니다.

Customer Journey Analytics에는 내장된 참여 세션 지표나 차원이 없습니다. 사용자는 비즈니스에 맞게 참여를 정의합니다. Adobe에서는 참여 기준을 캡처하는 세그먼트를 만들고 참여가 중요한 위치에 해당 세그먼트를 다시 사용할 것을 권장합니다. 관리자는 이 정의를 모든 사용자가 사용할 수 있도록 데이터 보기의 지표로 승격시킬 수도 있습니다.

자체 기준을 구체화할 때 사이트에 대한 값을 실제로 나타내는 신호를 선택합니다. 참여를 위한 세 가지 공통 구성 요소는 다음과 같습니다.

* **기간**: 최소 세션 길이(예: 10초 이상)
* **깊이**: 최소 이벤트 또는 페이지 보기 수(예: 2개 이상)
* **Action**: 등록 또는 구매와 같은 전환 또는 키 이벤트가 있습니다.

한 조건(GA4의 방식)을 충족하면 세션이 참여 중인 것으로 계산되도록 `OR`과(와) 결합하거나 보다 엄격한 요구 사항을 위해 `AND`과(와) 결합할 수 있습니다. GA4와의 패리티가 목표라면 기본값에서 시작하여 조정하십시오.

### 참여율

참여 세션 정의가 생성되면 참여 비율은 참여 세션 점유율입니다. 계산된 지표로 작성하려면 다음 작업을 수행하십시오.

1. Analysis Workspace에서 지표 목록 근처에 있는 **[!UICONTROL +]** 아이콘을 선택하여 계산된 지표 빌더를 엽니다.
2. 이름을 &quot;참여 비율&quot;로 지정하고 형식을 **[!UICONTROL 백분율]**(으)로 설정하십시오.
3. **[!UICONTROL 세션]**(으)로 나눈 참여 세션 세그먼트로 공식을 정의합니다.
4. **[!UICONTROL 저장]**&#x200B;을 선택합니다.

### 바운스 비율

GA4에서 바운스는 참여 세션의 역이므로 GA4의 바운스 비율은 `1 - Engagement Rate`과(와) 같습니다. 해당 공식을 사용하여 Customer Journey Analytics에서 두 번째 계산된 지표로 빌드합니다.

Customer Journey Analytics은 기본 제공 **[!UICONTROL 바운스 비율]** 지표도 제공하지만 기본 정의는 다릅니다. 단일 이벤트만 기록된 세션을 계산하며, 이는 많은 사이트에 대한 GA4의 정의와 반대 방향입니다. GA4의 바운스 비율을 `1 - Engagement Rate` 계산이 아닌 기본 [!UICONTROL 바운스 비율] 지표와 비교하면 숫자가 크게 달라집니다.

>[!TIP]
>
>Customer Journey Analytics의 세션 정의는 데이터 보기별로 구성할 수 있습니다. 해당 패리티가 조직에 대한 보고 요구 사항인 경우 바운스 및 참여 정의를 GA4의 기준(10초 기간, 2개 이상의 페이지 보기 또는 주요 이벤트)과 일치하도록 조정할 수 있습니다.

## 세션

GA4와 Customer Journey Analytics은 모두 기본적으로 30분의 비활성 시간 제한으로 설정되며 둘 다 자정 이후와 중간 세션 캠페인 변경 기간 동안 세션을 유지합니다. (Universal Analytics는 이러한 두 경우 모두에서 세션을 재설정하므로 일반적인 혼동의 원인이지만 GA4와 Customer Journey Analytics 간의 차이는 아닙니다.) 서로 다른 규칙은 다음과 같습니다.

| 규칙 | GA4 | Customer Journey Analytics |
|---|---|---|
| 비활성 시간 제한 | 속성 전체 조정 가능(기본 30분, 최대 7시간 55분) | 데이터 보기별 구성 가능 |
| 세션 시작 이벤트 | `session_start`만(자동) | 구성 가능, 모든 이벤트에서 세션 시작 가능 |
| 세션 종료 이벤트 | 없음 | 구성 가능, 모든 이벤트로 세션 종료 가능 |

Customer Journey Analytics의 세션 정의는 구성 가능하므로 세션 수는 데이터 보기가 설정된 방식에 따라 다릅니다. 데이터 보기의 시간 제한 및 세션 시작 이벤트를 GA4 속성에 일치시키면 플랫폼 간에 세션 수가 더 가깝게 계산됩니다.

## 사람 및 활성 사용자

GA4의 기본 사용자 지표인 **활성 사용자**&#x200B;는 날짜 범위에 참여 세션이 하나 이상 있는 사용자를 계산합니다. Customer Journey Analytics의 **[!UICONTROL 사람]** 지표는 날짜 범위에서 고유한 개인 ID를 계산합니다.

다음 몇 가지 이유로 이러한 지표가 다를 것으로 예상합니다.

* **참여 임계값**: GA4 활성 사용자는 [참여 세션](#engaged-sessions)이 없는 방문자를 제외합니다. Customer Journey Analytics의 [!UICONTROL 사람] 지표는 참여 수준에 관계없이 모든 사람을 포함합니다.
* **[!UICONTROL 결합]**: 결합이 활성화된 경우, 모바일 장치와 데스크탑에서 모두 방문한 사람은 Customer Journey Analytics에서 1명으로 계산되지만 GA4에서는 2명의 사용자로 계산될 수 있습니다. 일반적으로 결합을 통해 결합된 데이터 세트의 [!UICONTROL 사람] 지표가 GA4 사용자보다 낮습니다.
* **ID 모델**: GA4는 계단식 ID 모델을 사용하며, Customer Journey Analytics은 데이터 세트에 정의된 모든 사용자 ID를 사용합니다. 이러한 차이는 결합과 관계없이 개인 수에 영향을 줍니다.

## ID 및 결합

GA4는 계단식 ID 모델을 사용하여 사용자를 식별합니다.

1. User-ID(구현에서 설정한 경우)
2. Google 신호(사용자가 개인화가 활성화된 Google 계정에 로그인한 경우)
3. 장치 ID(쿠키 기반 클라이언트 ID)

대부분의 구현에서 해당 개인 ID는 ECID(Experience Cloud ID)입니다. 그런 다음 선택적 **[!UICONTROL 결합]** 기능을 사용하면 필드 기반 또는 그래프 기반 방법을 사용하여 크로스 디바이스 및 크로스 채널 ID를 확인하여 모바일 앱 세션과 데스크톱 브라우저 세션을 동일한 사용자와 연결할 수 있습니다.

ID 확인은 플랫폼마다 다르므로 사용자 수준 수가 정확히 일치하는 경우는 거의 없습니다. 이러한 불일치는 예상되며 데이터 품질 문제를 나타내지는 않습니다.

## 속성

GA4는 데이터 기반 속성을 기본값으로 사용하여 속성 수준(관리자 아래)에 구성된 보고 속성 모델을 적용합니다. Customer Journey Analytics과 마찬가지로 GA4는 보고서 시간에 이 모델을 평가하므로 이 모델을 변경하면 내역 및 향후 보고서가 소급하여 업데이트됩니다. 그러나 GA4에서는 모델이 속성 전체에 적용되며 이벤트 범위 트래픽 차원(예: Source, Medium 및 Campaign)을 사용하는 주요 이벤트 보고서에만 영향을 줍니다.

Customer Journey Analytics도 보고서 시간에 속성을 적용하지만 더 세분화된 제어를 사용합니다. 이를 구성하는 두 가지 위치가 있습니다.

* **데이터 보기 설정**: [속성 모델](/help/data-views/component-settings/attribution.md)을(를) 데이터 보기의 모든 지표 구성 요소에 설정하여 모든 보고서에서 해당 지표에 대한 기본값을 설정할 수 있습니다. 기본적으로 속성 모델이 적용되지 않습니다. 서로 다른 기본 속성 모델을 사용하여 동일한 지표의 여러 복사본을 포함하도록 데이터 보기를 구성할 수 있습니다.
* **구성 요소 수준 재정의**: 지표를 [!UICONTROL 자유 형식 테이블]&#x200B;(으)로 드래그한 후 열 헤더를 마우스 오른쪽 단추로 클릭하고 **[!UICONTROL 기본값이 아닌 속성 모델 사용]**&#x200B;을 선택하여 해당 인스턴스에 대해 재정의합니다. 직접 병렬 비교에 대해 서로 다른 속성 모델을 사용하여 동일한 지표를 테이블에 여러 번 드래그할 수도 있습니다.

GA4는 기본적으로 데이터 기반 속성으로 설정되고 Customer Journey Analytics은 모델을 구성하지 않으면 적용되지 않으므로 이들을 정렬하기 전까지 전환 및 채널 지표가 다를 수 있습니다. GA4를 마지막 클릭 모델로 설정하고 Customer Journey Analytics에서 일치하는 마지막 터치 모델을 구성하는 것은 like-for-like 기준선을 설정하는 가장 안정적인 방법입니다. Customer Journey Analytics의 모든 모델 변경 사항은 재처리 없이 모든 이전 데이터에 소급하여 적용됩니다.

## 불일치 감사

숫자가 예상보다 많은 경우 세 가지 감사 경로를 사용할 수 있습니다.

* **Assurance**: Adobe의 제품 내 유효성 검사 도구에서 XDM 이벤트가 올바르게 실행되어 Edge Network에 도달하고 Platform 데이터 세트에 기록되고 있는지 확인합니다. 보고서 번호를 비교하기 전에 이 도구를 사용하여 구현을 확인하십시오.
* **데이터 세트 미리 보기**: Platform UI에서는 모든 데이터 세트에서 원시 행을 미리 볼 수 있습니다. 이를 GA4의 DebugView 또는 BigQuery 내보내기와 비교하여 필드 수준의 정확도를 확인합니다.
* **Adobe Consulting**: 설명되지 않는 불일치가 지속되는 경우 Adobe 계정 팀에서 Adobe 컨설턴트와 공식적인 구현 감사를 예약할 수 있습니다.
* **수집 검토**: 보고 정의가 아니라 GA 데이터를 플랫폼으로 가져오는 방법에서 차이가 발생한다고 생각되면 [Google Analytics에서 데이터 마이그레이션](/help/use-cases/third-party/ga/overview.md)에서 수집 설정을 검토하십시오.
